# 2026 春 数理逻辑大作业

## 一、项目概述

本次课程设计考察两项核心能力：对复杂环境进行**逻辑形式化建模**的能力，以及基于 **Lean 的形式化验证**能力。具体围绕三个目标展开：(1) 将游戏环境抽象为逻辑结构；(2) 设计可解释的规划策略；(3) 使用 Lean 证明策略满足指定性质。

测评环境是一款 zelda-like 地牢探险游戏，各组需完成三件相互衔接的工作：

- **环境形式化**：将 Python 模拟器中的对象、属性、状态转移函数与交互规则，抽象为 Lean 中的数据类型、函数与谓词，并保证与 Python 环境语义一致；
- **策略实现**：在 Python 中实现能完成关卡目标的 Agent，**仅依据图像帧 `observation_mode = "pixels"` 进行决策**，实现方式开放（基于规则的策略、搜索算法、强化学习训练子任务 agent 等均可）；
- **性质证明**：在 Lean 中证明形式化环境与所设计策略满足要求的性质（安全性、可达性 / 完备性等）。

**分组**：5 人一组。环境仅提供原始图像帧及必要的位置信息，所需抽象由各组自行构建；每关附可参考的奖励（reward）模板供自行选用。

## 二、任务详细说明及具体要求

对指定关卡的环境与任务进行形式化建模，在此基础上设计能完成任务的策略，并在 Lean 中完成相关性质证明。

**1. 模拟器环境中的规划求解**
 在指定关卡中设计并实现能完成关卡目标的 Agent，可采用基于规则的策略、搜索算法、强化学习训练子任务 agent 等方式。Agent 须根据图像帧 `obs` 完成感知与决策。

**2. Lean 中的形式化与证明**
 在 Lean 中实现对关卡的形式化建模，包括对象、属性、函数（状态转移）、交互规则等；将 Python 实现中的环境设定与策略逻辑抽象为 Lean 中的数据类型、函数、谓词，确保语义一致；并在此基础上完成要求的性质证明。

## 三、任务环境
💡 有关测试环境的详细说明 [Task环境说明](task.md)

模拟器为塞尔达风格的迷宫冒险环境，每个房间为 **160 pixel × 128 pixel**（仅地图部分），关卡间房间可拼接，移动以像素为单位，单次执行步数可在接口中调节。

**玩家角色**

- 状态属性：位置、生命值、金币、钥匙、背包等。
- 动作操作：四个移动操作（玩家朝向为最后移动方向）和两个行动操作——使用剑 `slot_A`（攻击范围为面向方向前方一格）、使用盾 `slot_B`（可格挡伤害）。
- 血量与资源：初始血量为 5，触碰陷阱或怪物会减少血量；金币、钥匙等可通过开启宝箱或击败怪物获得。

**可交互对象**

- **宝箱 chest**：阻挡移动，开启后提供奖励（钥匙、金币等）。
- **陷阱 trap**：接触造成伤害，然后回到房间的初始位置。
- **按钮 button**：站上触发，可满足特定出口条件。
- **拉杆 switch**：按下 `BUTTON_A`（键盘对应为'z'）可切换地图状态。
- **怪物 monster**：移动型敌人（chaser / patroller / ambusher 三种类型），接触玩家造成伤害，可被剑攻击击杀，击杀奖励金币。

**地图与观测**：观测以像素图像 `obs` 给出；出口与任务条件依各关卡设定。

## 四、关卡设计

共 **5 个关卡**，难度递增（两简单、两中等、一更难）。

- **关卡 1**：单房间，取钥匙开门。
- **关卡 2**：单房间，击杀怪物。
- **关卡 3**：多房间任务链（约三房间）。
- **关卡 4**：五房间结构，中央桥接房需按按钮切换连通方向，玩家初始无剑，依次完成"取钥匙 → 开宝箱取剑 → 击杀小怪 → 开启胜利宝箱"。
- **关卡 5**：综合关卡，融合多种机制，目标为通关；鼓励在报告中说明你对该关可达性的分析。

## 五、评分细则（满分 100%）

| 模块             | 占比 |
| ---------------- | ---- |
| 环境形式化       | 30%  |
| 策略形式化与证明 | 30%  |
| 策略性能         | 30%  |
| 报告与可读性     | 10%  |

**模块一　环境形式化（30%）**
 考察 Lean 形式化是否正确、完整、无冗余：状态 / 动作 / 对象 / 目标谓词定义是否齐全；动作转移语义（移动、撞墙、踩陷阱、开箱、攻击、按钮切换等）是否合理；是否证明基本安全性或不变量（如合法移动不出界、不进墙）。

**模块二　策略形式化与证明（30%）**
 考察策略的可验证性。硬编码策略需形式化 planner / 搜索 / action mask 逻辑，并证明其输出合法、安全、成功时满足目标，可进一步证明完备性（"若存在可行路径则必能找到"）；机器学习策略需明确 Lean 证明覆盖的可验证层（如 action mask、safety shield、符号 planner、轨迹验证器、奖励谓词），并证明经该层约束后的行为满足规范。

**模块三　策略性能（30%）**
 通过统一的**一键测评脚本**自动评测（仅做推理，多次并行运行统计完成关卡数）。未完全通关但取得阶段性进展（到达中间目标、击杀小怪、获取关键道具等）按完成程度计中间分，可在报告中说明所达成的子目标。

 测评脚本的 policy 接口、命令行参数、JSON 输出格式，以及 `grayscale`、`redraw_geometric`、`redraw_symbols` 等图像鲁棒性测试方法，见 [测评脚本说明](evaluation.md)。

 ⚠️ 要求：
 - 以黑盒方式调用提交的 Agent（输入 obs，info，输出动作序列），与测评环境接口对齐；
 - 统计每关的成功率（完成关卡目标的 episode 占比），并以此为准评分；报告中所述性能须与测评结果一致，明显不符或无法复现按学术诚信处理。
 - Agent 应为 python 脚本格式

**模块四　报告与可读性（10%）**
 报告是否清晰详细，说明形式化思路、策略设计、所证定理与实验结果；Lean 代码命名与证明结构是否清晰；是否说明模型假设与简化之处。

**加分项**（计入后总分不超过 100）：使用强化学习 / 机器学习并实现较高自动化程度；采用统一策略覆盖多关卡、展现泛化能力或结构复用（优于每关单独训练或单独硬编码）；证明了基本要求之外、更具难度或价值的定理。实现方式不影响是否计分，**不对任一方式扣分**，仅以加分制鼓励自动化与泛化。

## 六、提交要求与真实性

以 zip 提交，须含：报告（含所证定理列表与测评结果）、Lean 源文件、完整 Python 源代码（强化学习须附**模型权重**并与测评接口对齐，按指定接口输出动作序列供黑盒调用）、**能证明代码确实运行的截图或附件**。

- 报告所述方法与成功率须与提交的代码 / 权重一致；性能以测评脚本实际结果为准，明显不符或无法复现按学术诚信处理。
- Lean 文件须通过检查，**不允许未说明的 `sorry` / `admit` / `axiom`**。
- 若简化了真实环境，须在报告中说明；使用机器学习时不要求证明网络本身永远正确，但须指明 Lean 证明覆盖的可验证层。
